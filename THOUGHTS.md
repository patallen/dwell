# Neurodivergent Productivity: Research Reference

*Distilled from three research iterations, 2026-03-07. See [sources](#sources) at end.*

---

## The Core Problem

Traditional productivity assumes the brain converts **importance** into **action**. The ADHD brain doesn't work this way. It runs on an **interest-based nervous system** (Dodson) — activating on **PINCH**: Passion, Interest, Novelty, Challenge, Urgency. When those ingredients are present, it excels. When absent, no willpower bridges the gap. This is neurochemistry (lower prefrontal dopamine availability), not character.

ADHD is not an attention *deficit* — it's an **attention regulation disorder**. Total attention may equal or exceed neurotypical; it's the *distribution* that's uncontrollable without external support. The same mechanism that prevents starting a 10-minute email enables 8 hours of coding without eating.

**The neurodivergent brain is a different operating system, not a broken one.** The problem is environmental mismatch — measuring productivity with a ruler designed for a different shape. "Neurocognitive Mismatch Theory" (2025) argues ADHD traits were likely adaptive in ancestral contexts; disability is substantially created by modern environmental demands.

## Key Mechanisms

### Emotional Dysregulation (Core, Not Comorbid)

Emotional dysregulation flows through the same prefrontal/amygdala/striatal structures as executive dysfunction — it's not a side effect, it's the same broken pipes (Barkley).

**The shame-procrastination spiral:** Negative feeling → avoidance → self-criticism → higher activation threshold → repeat. Compounded by temporal processing deficits (any emotion can feel permanent) and **Rejection Sensitive Dysphoria** (RSD degrades the cognitive machinery needed for task initiation).

**Implication:** Any system that ignores emotional state is building on sand.

### The Wall of Awful (Mahan)

Accumulated task-associated negative affect. Built of "failure bricks" starting ~4th grade. When approaching a task, you face its entire emotional history. Why someone can learn a language in a weekend but can't make a phone call.

Response patterns: freeze (staring at wall), avoidance (going around), anger-powered breakthrough (Hulk Smash — effective short-term, destructive), or **putting a door in the wall** (changing emotional state before approaching: music, movement, comedy, connection).

### Working Memory: The 30% Tax

~30% developmental delay in executive functions (Barkley). The mental whiteboard is smaller — context falls off during transitions. This may be *upstream* of inhibitory control problems (2024 Frontiers). Each interruption overwrites previous context on a whiteboard too small to hold both.

**Cognitive offloading** (Risko & Gilbert 2016) is the research-backed response: externalize information processing to reduce cognitive demand. But there's a **bootstrapping problem** — second-brain systems often require the executive function they're designed to compensate for. *The scaffolding cannot require the function it scaffolds.*

### Hyperfocus vs. Flow

Critical distinction (Ashinoff & Abu-Akel 2021): **negative correlation between hyperfocus and flow on perceived control.** In flow, you control engagement. In hyperfocus, the activity controls you. Flow has self-regulation intact; hyperfocus has it suspended.

Engineering flow for ADHD: challenge-skill calibration (narrow sweet spot), immediate feedback loops, self-chosen goals (SDT autonomy), time-bounded sessions with external exit ramps, environmental control.

**Design tension:** Features enabling deep focus (blocking interruptions) also enable destructive hyperfocus (no exit ramps). Optimize for *regulated* engagement.

### Two Cognitive Modes

1. **Convergent mode** — sustained linear focus. The leaky DMN (Default Mode Network, which doesn't deactivate properly in ADHD) is a liability. Use external scaffolding, body doubling.
2. **Divergent mode** — creative problem-solving. The leaky DMN is an *asset*. Create low-demand conditions (walking, doodling) that let it run while steering output.

Most productivity systems only optimize for convergent mode.

### Autistic Inertia + ADHD Paralysis (AuDHD)

Different mechanisms that compound: ADHD paralysis is dopamine-mediated (task not interesting enough). Autistic inertia is about enormous activation energy for state changes regardless of interest. Together: can't start because it's not interesting enough *and* can't stop current activity because transitions are painful.

### Circadian Misalignment

ADHD has a circadian component: 78% show delayed sleep-wake timing, melatonin onset delayed ~90 min in adults. When testing aligns with actual chronotype, executive function is **significantly enhanced** (Frontiers in Psychiatry 2025). Forcing early-morning work fights ADHD *and* circadian misalignment simultaneously.

### Interoception Deficits

Reduced ability to sense hunger, thirst, fatigue, pain — correlated with ADHD symptom severity (2025 systematic review). Creates hidden compounding: hyperfocus → miss body signals → depletion → executive function degrades → crash → interpret as laziness. Simple timed prompts (water, food, movement) may outperform any task management feature.

### Demand Avoidance (PDA Spectrum)

Resistance to demands including self-imposed ones. Goal-setting creates demands → avoidance. To-do lists become lists of demands to resist. Even wanting something, once it becomes obligation, triggers avoidance. **The most effective strategies make structure feel like freedom.** Suggest, don't command.

### The 2e Trap (Twice-Exceptional)

Giftedness masks disability, disability masks giftedness. Peak performance becomes "proof" of consistent capability → gap interpreted as moral failure → "you're so smart, why can't you just..." → shame → Wall grows. **Never use past performance as baseline.** Variability IS the condition.

### Motivation: Play Mode vs. Work Mode (SDT)

Self-Determination Theory (Deci & Ryan): autonomy, competence, relatedness. ADHD is not a motivation deficit — it's a deficit in motivation for *externally imposed* goals (Morsink 2022). Play mode (intrinsic) uses the strong pathway; work mode (extrinsic) uses the weak one. Gamification can *undermine* intrinsic motivation via overjustification effect.

### Medication as Floor-Raiser

MTA Study: medication + behavioral > behavioral alone for core symptoms. Medication normalizes task-switching costs to neurotypical levels. But at 8-year follow-up, treatment groups no longer differed — medication creates conditions for learning skills, doesn't teach them. Without medication, scaffolding needs to be heavier.

### Structured Procrastination (Perry)

Procrastinators do marginally useful things, not nothing. Structure this: place nominally important items at top knowing they'll be avoided; items actually completed lower down are genuinely worthwhile. Breaks down in severe paralysis where everything is equally aversive.

## What Actually Works

Core principle: **change the environment, not the person.**

- **Movement before cognition** — raises dopamine/norepinephrine to cross activation threshold
- **Novelty rotation** — structured variety, not chaotic switching
- **Urgency engineering** — artificial deadlines, timers, public commitments (balance: anxiety threshold → burnout)
- **External scaffolding at point of performance** — reminders where/when decisions happen
- **Flexible structure** — autonomy over *how* work gets done
- **Activation work** — movement, environment setup, sensory regulation (real productivity, not procrastination)
- **Recovery as infrastructure** — ND brains burn more fuel on executive tasks; rest isn't optional

## The Productivity Stack

Hierarchy of conditions for neurodivergent productivity (each layer depends on those below):

| Layer | Domain | Key Principle |
|---|---|---|
| **0** | **Physiological baseline** | Sleep (aligned to chronotype), food, water, movement. Interoception deficits mean proactive external monitoring — don't wait for signals that won't come. |
| **1** | **Emotional state** | Is the Wall manageable? Active RSD/shame spiral? Activation work (music, movement, connection) operates here. Never baseline against past peaks (2e trap). |
| **2** | **Environmental scaffolding** | External supports at point of performance. This is the core intervention, not a workaround. Cognitive offloading primary strategy, but must not require the function it scaffolds. |
| **3** | **Task-interest alignment** | PINCH criteria. Reframe, break down, rotate for novelty. Distinguish flow-conducive vs. hyperfocus-conducive conditions. Build in exit ramps. |
| **4** | **Temporal alignment** | Right time of day for this cognitive type. Transition awareness — every switch carries double penalty (direct cost + attention residue). Provide context restoration. |
| **5** | **Autonomy preservation** | SDT: autonomy, competence, relatedness. Play mode over work mode. System suggests; user decides. Structure that feels like freedom. |
| **6** | **Medication awareness** | Medication raises floor, normalizes transition costs. Without it, heavier scaffolding needed. |

Most productivity tools only operate at Layer 3. They fail because the foundation isn't addressed. **Work the whole stack.**

## The One Insight

**The scaffolding cannot require the function it scaffolds.**

Every failed ADHD system fails for the same reason: it requires executive function to compensate for executive function deficits. Successful interventions (body doubling, environmental design, medication, simple cues at point of performance) impose zero or near-zero executive function overhead.

The tool that works is the one that works on your worst day, not your best day.

## Product

### What adhdeez Is

adhdeez is a productivity tool built *from* the research above, not from neurotypical productivity assumptions. It's an environmental adaptation — reshaping the immediate digital workspace to fit the ADHD/AuDHD brain that's actually doing the work.

**Core metaphor: a context stack, not a to-do list.** The brain doesn't work in flat lists. It works in nested contexts — you were doing X, then Y interrupted, and now you need to get back to X but you've lost the thread. The context stack makes this explicit: push what you're working on, handle the interruption, pop back. The tool remembers so the working memory doesn't have to.

### Core Model

**Everything is a note.** A note is the single top-level entity. It has a type that determines how the app treats it.

**Note types:**
- **Project** — has a goal, has a status, ends when it's done
- **Meeting** — tied to a point in time, captures what happened and what to do next
- **1:1** — recurring, tied to a person, accumulates over time
- **Research / Area** — ongoing thinking about a topic, no end state

**A note can contain:**
- Body text (rich text — planning, thinking, reference, whatever)
- Child notes (sub-notes — research findings, sub-topics, discrete items)
- Questions (things to figure out — guide planning and research)
- Tasks (things to do — what the focus engine operates on)

**Notes are nested.** A sub-note is just a note with a parent. You create it in context and it belongs to its parent. You don't name files, pick folders, or organize anything. The app handles storage. The hierarchy is implicit from where you created things.

**You find things by searching, not browsing.** RAG + fuzzy search means you don't need to remember where something is or what you called it. Just what it was about. The organizational structure is invisible to the user.

**The focus engine surfaces what needs attention.** Currently it only surfaces tasks. It should surface notes too — a project with unanswered questions needs attention, a meeting note with open action items needs attention. The type determines how something is scored and presented.

**The app does two things:**
1. **Helps you think** — you open a note and work through it. Questions keep you on track. Sub-notes capture what you learn. Tasks fall out of that process.
2. **Helps you act** — the focus engine looks across everything and tells you what needs attention next.

Right now the app only does #2, and poorly, because there's no surface for #1. There's nowhere to think, so there are no well-formed tasks, no questions, no context. The engine is starved.

**How this differs from Obsidian:** Obsidian is a blank canvas — you impose structure, you organize files, you maintain links. That's executive function overhead. adhdeez has opinions. It knows what type of note you're in and helps you based on that. Organization is the app's job. Finding things is search, not navigation.

### Architecture Decision: Merge Python Backend into Tauri/Rust

**Decision: Yes. The Python daemon should be replaced by Rust handlers inside Tauri.**

**Why:**

1. **Single binary distribution.** No Python dependency, no venv, no daemon management. Download → open → done. Installation friction kills personal tools.

2. **Native system integration.** Running as a Tauri app unlocks:
   - **System tray** — lives in the menu bar, always available, no visible window needed
   - **Launch on login** — registers as a login item natively
   - **Global shortcuts** — capture from anywhere (⌘+Shift+A) without switching apps
   - **Active window tracking** — know what app/file the user is in (feeds knowledge graph and context)
   - **File watching** — detect file changes during focused tasks (context restoration)
   - **Native notifications** — body prompts and session guardrails as system notifications, not just in-app

3. **Tauri IPC instead of HTTP.** Frontend calls `invoke("get_focus", { energy: "calm" })` directly into Rust. No ports, no "is the daemon running?", no network overhead.

4. **The Python backend is thin.** What it does today:
   - File I/O: read/write Markdown + YAML frontmatter → `serde_yaml` + `std::fs`
   - CRUD on tasks/projects/questions → straightforward Rust structs
   - Suggestion scoring algorithm → pure logic, easy to port
   - Context stack → JSON file read/write → `serde_json`
   - No heavy Python dependencies, no async complexity, no ML libraries

5. **Background operation model.** The app becomes a persistent background service:
   - System tray icon with quick actions (capture, show window, pause)
   - Body prompts as native notifications even when window is closed
   - Session timer runs regardless of window visibility
   - Window opens/closes freely without losing state
   - Global shortcut to summon the window from any context

**Migration approach — incremental, not big-bang:**
1. Implement new features (M7+) as Tauri commands in Rust alongside existing Python API
2. Add system tray + background operation (this is high-value and independent of the port)
3. Port existing endpoints one by one: tasks → projects → questions → context → focus
4. Remove Python daemon once all endpoints are ported
5. Existing frontend `fetch()` calls become `invoke()` calls — mechanical refactor

**Knowledge graph / RAG strategy:**
- Start with keyword matching in Rust (fast, no dependencies)
- For semantic search, call Claude API or embeddings API from Rust (`reqwest` HTTP client)
- If a local model is needed later, consider ONNX Runtime (has Rust bindings) or a small Python sidecar just for inference
- Don't let the RAG question block the port — it's a later feature on a different timeline

**What this enables that's currently impossible:**
- "You've been in VS Code for 2 hours editing auth files" → auto-associate with the auth task
- Capture from anywhere without switching to the app
- Body prompts as system notifications when the app window isn't open
- True "runs on startup, always there" experience like a system utility, not an app you have to remember to open

### How the Research Maps to Features

| Research Insight | Existing Feature | Gap / Opportunity |
|---|---|---|
| **Working memory deficit / cognitive offloading** | Context stack (push/pop), persistent state | Context restoration at transitions — "here's where you were, what you were doing, what was next" |
| **Interest-based nervous system (PINCH)** | Focus suggestions | Could surface *why* a task might be engaging right now (novelty, challenge, urgency) rather than just what to do |
| **Wall of Awful** | — | Emotional check-in before task presentation. Never guilt. Never compare to past performance. |
| **Interoception deficits** | — | Proactive body-state prompts: water, food, movement, posture. Timer-based, not feeling-based. |
| **Circadian alignment** | — | Learn when the user actually works well. Present hard tasks during peak windows, divergent/creative tasks during off-peak. |
| **Hyperfocus vs. flow** | — | Session guardrails: gentle exit ramps after extended focus. "You've been going for 2 hours — water? stretch?" Not a nag; a door in the wall. |
| **Autonomy preservation / PDA** | Keyboard-driven UI, no forced workflows | Keep this. Never mandate. Suggestions, not commands. "You might want to..." not "You need to..." |
| **Structured procrastination** | Task list with projects | Could lean into this: surface alternative productive tasks when the main task feels blocked, without judgment |
| **Transition costs / attention residue** | Context stack | Strengthen: when popping context, show a brief "re-orientation" — what was the state of this task when you left it? |
| **Bootstrapping problem** | Lightweight capture (⌘I) | Critical to maintain: zero-friction entry points. The system must work on your worst day. |
| **2e trap** | — | Never display streaks, averages, or comparisons. Variability is the condition. Celebrate today's wins without implying tomorrow should match. |

### Design Principles (from the research)

1. **The scaffolding cannot require the function it scaffolds.** Every feature must pass the test: "Can someone with depleted executive function still use this?" If no, simplify until yes.

2. **Work the whole stack, not just Layer 3.** Don't just manage tasks. Check the foundation: Are you hydrated? Have you moved? Is the emotional state workable? Is this the right time of day for this work?

3. **Optimize for regulated engagement, not just engagement.** Deep focus is good; uncontrolled hyperfocus is dangerous. Build in exit ramps without breaking flow.

4. **Never build the Wall higher.** No guilt. No streaks. No "you did X yesterday, why only Y today?" No shame spirals. The system is a ramp, not a judge.

5. **Play mode over work mode.** Frame everything as choice, not obligation. SDT: autonomy, competence, relatedness. The system suggests; the user decides.

6. **Environment adaptation, not behavior modification.** This is glasses for the mind, not a drill sergeant. The brain works differently; the tool adapts the environment to match.

### What's Built vs. What's Next

**Already built (v0):**
- Context stack with push/pop/remove — the core cognitive offloading primitive
- Focus state with smart suggestions — early interest-alignment
- Quick capture (⌘I) — zero-friction entry, respects the bootstrapping constraint
- Project → task hierarchy with questions system
- Rich text editor for task/project bodies
- Keyboard-first UI with overlays (find, stack, projects, help, settings)
- Persistent file-based storage (Markdown + YAML frontmatter in ~/.adhdeez)

**Next priorities (informed by gaps above):**

1. **Context restoration on transitions** — When popping the stack or switching tasks, show a brief re-orientation: last edit, open questions, next step. This directly addresses working memory + attention residue.

2. **Body-state prompts** — Timed, gentle, dismissible prompts for water/food/movement/posture. Not reactive to user state (we can't sense it); proactive on a schedule. Addresses interoception gaps.

3. **Emotional ground-floor** — Before presenting today's work, offer a low-key check-in. Not a mood tracker (that's a demand). Something more like: "How's it going?" with options that quietly adjust what gets surfaced. High Wall of Awful → surface novel/easy tasks. Low Wall → surface the harder stuff.

4. **Session guardrails** — After extended continuous use (configurable), surface a gentle prompt. Not an interruption — a notification that can be dismissed. "Been at it for 90 min. Water? Stretch? Save your place?" Addresses hyperfocus risk without breaking flow.

5. **Time-of-day awareness** — Track when tasks get completed (not started — completed, because that's what indicates real cognitive engagement). Over time, learn peak windows. Use this to inform focus suggestions. Don't display it as data; just let it quietly shape what gets surfaced when.

### UX Patterns & Implementation Sketches

#### 1. Context Restoration (Priority 1)

**The problem:** When you pop the context stack or switch tasks, your working memory whiteboard is blank. You've lost the thread.

**UX pattern — "Where Was I?" card:**
- On context pop or task switch, show a transient card (not a modal — no demand) with:
  - Task title + last few lines of the body/notes you edited
  - Open questions attached to this task/project
  - Time since you last touched it ("you left this 45 min ago")
  - One-line "next step" if you wrote one (freeform field on tasks, populated at push time)
- Card auto-dismisses after 10s or on any keypress. Not blocking. Just *there*.
- **Push-time prompt:** When pushing context (switching away), ask one optional question: "Quick note for when you come back?" — stored on the context entry. This is the "note to future self" that bridges the gap. Skip with Enter (zero friction if you don't want it).

**Data model change:** Add `note: str | None` to `ContextEntry`. Add `next_step: str | None` to `Task`.

**Implementation:** Frontend card component, ~50 lines. Backend: two fields on existing models. The cognitive load test: you can ignore all of this (dismiss/skip) and it still works.

#### 2. Body-State Prompts (Priority 2)

**The problem:** Interoception deficits mean you won't notice you're dehydrated, hungry, or stiff until you crash.

**UX pattern — ambient pulse:**
- A subtle visual element (gentle color shift on the status bar, or a small icon) that changes state on a timer. Not a popup. Not a sound. Just a quiet visual cue at the periphery.
- Default intervals: water (45 min), movement (60 min), meal anchors (4 hours since last dismissal).
- Single keypress to acknowledge ("got it" / "not now"). "Not now" delays 15 min, not permanently. No tracking, no streaks, no guilt.
- **Configurable in settings.** Can turn off entirely. Autonomy preservation.
- Resets on app focus (if you've been away from the app, you might have already gotten water).

**Implementation:** Frontend timer + subtle CSS animation. No backend changes needed — timer state is ephemeral (session-only). Settings stored in localStorage or vault config.

**The bootstrapping test:** Completely passive. Requires zero executive function to benefit from. The app does the noticing your body can't.

#### 3. Emotional Ground Floor (Priority 3)

**The problem:** If the Wall of Awful is high, no amount of task management helps. But a "mood tracker" is a demand.

**UX pattern — the soft landing:**
- On first app open of the day (or after 4+ hours away), show a single low-pressure screen before the task view. Not a form. Not a questionnaire. Something like:
  - Three options, visually distinct, no labels required: a calm color (good), a neutral color (meh), a stormy color (rough)
  - Single click/keypress. No explanation required. No data stored long-term.
- **What it does:** Adjusts the focus suggestion algorithm for this session:
  - Calm → normal suggestions, include harder/higher-wall tasks
  - Neutral → default mix
  - Rough → surface only: novel tasks, low-wall tasks, quick wins, or explicitly suggest activation work ("maybe start with a walk?")
- **What it never does:** Compare to yesterday. Show trends. Say "you've been rough 3 days in a row." No data accumulation. Today is today.

**Implementation:** Frontend gate component before main view. Backend: add `energy_level: str | None` param to `/focus` endpoint that adjusts suggestion scoring. No persistence.

**The autonomy test:** One click with genuine permission to skip (press Escape → default/neutral). Never feels like a demand.

#### 4. Session Guardrails (Priority 4)

**The problem:** Hyperfocus ≠ flow. You need exit ramps but not interruptions.

**UX pattern — the gentle wave:**
- After N minutes of continuous interaction (default: 90, configurable), surface a non-blocking notification. Not a modal. Not a sound. A subtle bar that slides in at the top or bottom:
  - "Been at it for 90 min. Water? Stretch? You're doing great."
  - Single keypress to dismiss. Or ignore — it auto-fades after 30s.
- Repeats every N minutes if dismissed (but with diminishing urgency — don't become the nag).
- **Never interrupts typing or active editing.** Waits for a natural pause (2s of no input).
- Option to "snooze for 30 min" — explicit autonomy.

**Implementation:** Frontend idle-detection timer + notification component. No backend.

#### 5. Time-of-Day Awareness (Priority 5)

**The problem:** The brain has a circadian profile. Presenting hard convergent tasks during a trough is setting up failure.

**UX pattern — invisible learning:**
- Track `completed_at` timestamps on tasks (already have `updated_at`).
- After ~2 weeks of data, cluster completion times. Don't display this. Let it quietly weight focus suggestions:
  - Tasks similar to ones completed during morning hours get boosted in morning suggestions
  - Creative/divergent tasks (tagged or inferred from project type) get boosted during historically off-peak times
- **Never show the user a chart of their "productive hours."** That's a 2e trap — they'll compare bad days to good days. The data stays invisible; its only expression is better suggestions.

**Implementation:** Backend: record `completed_at` on task status change. Add time-window weighting to suggestion algorithm in `store.py`. No frontend changes needed — the system just gets quietly smarter.

### Implementation Roadmap

Each milestone is independently shippable. Order reflects dependency + impact.

**M1: Context Restoration** (smallest change, highest research-backed impact)

Files changed:
- `daemon/models.py` — add `note: str | None = None` to `ContextEntry`, add `next_step: str | None = None` to `Task`
- `daemon/main.py` — accept `note` in push context request, return `note` in pop response
- `daemon/store.py` — persist `next_step` in task frontmatter, persist `note` in context.json
- `frontend/src/api.ts` — update types + push/pop calls
- `frontend/src/App.tsx` — add push-time "note to self" input (optional, inline), add "Where Was I?" transient card on pop
- New: `frontend/src/components/WhereWasI.tsx` — the restoration card

API changes:
```
POST /context/push  { ref_id, ref_type, reason, note? }
POST /context/pop   → returns { popped_entry, restored_entry? }  # restored = new top of stack
```

Estimated scope: ~2 hours. Zero new dependencies.

**M1 Review Notes:**

1. **The push `note` is the highest-value piece.** The WhereWasI card showing last-edited lines and open questions is useful, but the user's own words ("I was about to refactor the auth check") will always beat system-inferred state for context restoration. UX must make this feel effortless — not a form field.

2. **Drop or extend the auto-dismiss timer.** 10s is too short for re-orienting after a long interruption. Prefer dismiss-on-keypress (any key = "got it, I'm back") over timer-based. A timer creates subtle demand ("read fast"), which fails the bootstrapping test. If keeping a timer, make it 60s+.

3. **Consider deferring `next_step` on Task.** It overlaps with the push `note` — one is task-level, one is context-switch-level. Having both may confuse the UX. Start with just the push `note` and add `next_step` later if users want a persistent "what's next" independent of the stack. Simpler launch, easier to evaluate.

4. **API shape is clean.** Returning `restored_entry` on pop avoids a second round-trip. Good.

**M2: Body-State Prompts** (frontend-only, no backend)

Files changed:
- New: `frontend/src/components/AmbientPulse.tsx` — timer logic + visual cue
- New: `frontend/src/hooks/useBodyPrompts.ts` — timer management, visibility detection, snooze logic
- `frontend/src/App.tsx` — mount AmbientPulse, wire settings
- Settings stored in `localStorage` key `adhdeez:bodyPrompts` (intervals, enabled/disabled per type)

No API changes. No backend changes. Pure frontend timer + CSS.

Estimated scope: ~1.5 hours.

**M3: Emotional Ground Floor** (small frontend + backend param)

Files changed:
- New: `frontend/src/components/SoftLanding.tsx` — three-option check-in screen
- `frontend/src/App.tsx` — gate main view behind SoftLanding on cold start (use `sessionStorage` to show once per session)
- `frontend/src/api.ts` — pass `energy` param to focus endpoint
- `daemon/main.py` — accept `energy: str | None` on `/focus`
- `daemon/store.py` — adjust suggestion scoring based on energy level

API changes:
```
GET /focus?energy=calm|neutral|rough  # defaults to neutral if omitted
```

Suggestion adjustments:
- `rough` → boost tasks with no deadline, no parent (low-stakes), and tasks never attempted before (novel). Deprioritize overdue or high-loe tasks.
- `calm` → include higher-loe and deadline-approaching tasks in suggestions.

Estimated scope: ~2 hours.

**M4: Session Guardrails** (frontend-only)

Files changed:
- New: `frontend/src/components/GentleWave.tsx` — notification bar
- New: `frontend/src/hooks/useSessionTimer.ts` — tracks continuous interaction, waits for idle before showing
- `frontend/src/App.tsx` — mount GentleWave

No API changes. Timer + idle detection + CSS transition.

Estimated scope: ~1 hour.

**M5: Time-of-Day Awareness** (backend-only, invisible)

Files changed:
- `daemon/models.py` — add `completed_at: datetime | None` to `Task`
- `daemon/store.py` — set `completed_at` on status change to "done", add time-window weighting to suggestion algorithm
- `daemon/main.py` — no changes (completed_at set automatically)

No frontend changes. No API changes. The system silently gets smarter.

Estimated scope: ~2 hours (mostly in tuning the suggestion weights).

### What This Is Not

- Not a habit tracker (habits require consistency; consistency is the deficit)
- Not a gamified system (extrinsic rewards undermine intrinsic motivation)
- Not a calendar replacement (rigid time-blocking fights the brain)
- Not a "second brain" that requires a PhD to maintain (bootstrapping problem)
- Not a system that works great on your best day and fails on your worst

**It's a tool that works on your worst day.** That's the bar. Everything else is a bonus.

## M6+: What's Missing

### Unsolved Problems

1. **Can't plan or research a project** — Projects have a body and questions, but questions aren't focusable. You can't push a question onto the stack, research it, write notes on it, answer it, and pop back. Planning a project means starting with questions and uncertainty — the answers produce tasks. The app doesn't support this flow.
2. **Can't edit tasks** — Body, LoE, deadline, project assignment all supported by the API but not editable from the UI.
3. **Can't take notes while working** — Task body is read-only in the focus view. No way to capture what you're learning or deciding as you work.
4. **Can't break tasks into steps** — No subtask UI. The `parent` field exists (points to projects) but there's no task-to-task hierarchy.
5. **Can't create tasks with properties** — Quick Capture sends only `{ title }`. The suggestion algorithm scores on deadlines and LoE that users can't set.
6. **Completed tasks vanish** — No way to see what you accomplished.
7. **Can't browse/filter tasks** — Search finds by name only. No filtering by status, project, LoE, deadline.
8. **Can't skip a task gracefully** — Only options are do it, drop it, or close the app.

### M6: Focusable Questions (Project Planning & Research)

**Problem:** You're trying to plan a project. You have uncertainty — things you need to figure out before you can act. The app has no way to guide you through that process.

**What this changes:** Questions become focusable, same as tasks. They're a valid `ref_type` in the context stack.

**The flow:**
1. You focus on a project
2. You see its open questions (or create new ones)
3. You push a question onto the stack — "Which auth library should we use?"
4. You're now focused on that question. You can write notes as you research (the question gets a `notes` field, rich text, same TipTap editor as project body)
5. A new question comes up during research — push it onto the stack. Answer it. Pop back.
6. When you answer the question, the answer can produce tasks — "Use Passport.js" → create task "Set up Passport.js auth"
7. Pop back to the project. Next unanswered question surfaces.

**What this adds to the data model:**
- `Question` gets a `notes: str` field (rich text for research notes, separate from the final `answer`)
- Context stack accepts `ref_type: "question"` with `ref_id` pointing to a question
- Focus endpoint handles questions: when focused on a question, show the question text, notes editor, answer field, and the parent project context

**What this adds to the UI:**
- Question focus view: question text at top, TipTap notes editor, answer input, parent project context, related tasks
- "Create task from this" action when answering a question
- Questions appear in the project focus view as a list you can push onto the stack

**Why this is M6:** The app already handles "what should I do?" (task focus + suggestions). It doesn't handle "what do I need to figure out?" — which comes first. You plan before you execute. Questions are the planning primitive. Making them focusable extends the core mechanic (focus + stack) to cover the full workflow: research → plan → execute.

**Scope: ~4h.** Data model change is small. Focus view for questions reuses existing patterns. The context stack already handles different ref types.

### M7: Editable Focus View

**Problem:** Task body, LoE, deadline, project aren't editable from the UI.

**What this changes:** The focus view becomes editable instead of read-only.

- Title: click to edit inline
- Body: TipTap editor (reuse from ProjectView), always editable. Currently renders as `<p>{task.body}</p>` which will break on HTML content — replace with TipTap.
- LoE: clickable dot, cycles hot/warm/cool/none
- Deadline: click to show date input
- Project: click to assign via dropdown
- All auto-save on blur via existing `updateTask()` API

**Scope: ~3h.**

---

### M8: Subtasks

**Problem:** Can't break tasks into steps.

- New `parent_task: str | None` field on Task (separate from `parent` which = project)
- In focus view: subtask list below the body, inline creation, checkbox to complete
- Subtasks inherit project from parent task
- Subtasks appear in suggestions independently
- "Create task from this" on question answers (from M6) uses the same mechanism

**Scope: ~3h.**

---

### M9: Enhanced Quick Capture

**Problem:** ⌘+I sends only `{ title }`.

- Default: title-only, Enter to save. Unchanged.
- Tab after typing → expands: LoE toggle, deadline, project dropdown. All optional.
- Enter at any point saves with whatever's filled in.

**Scope: ~1.5h.**

---

### M10: Completion History

**Problem:** Completed tasks vanish.

- ⌘+D overlay: completed tasks grouped by day
- Task title, project, completion time
- Can reopen if completed by accident
- No charts, no comparisons

**Scope: ~2h.**

### M11: Filtered Browsing

**Problem:** Can't browse tasks structurally.

- Enhance ⌘/ with filter toggles: status, project, LoE, overdue
- Sort by: created, deadline, LoE
- Keyboard-driven

**Scope: ~2.5h.**

### M12: "Not Now" Action

**Problem:** Can't skip a task without dropping it.

- `n` on focused task: deprioritizes it for this session, surfaces alternatives
- No tracking, no guilt, resets on next session

**Scope: ~1.5h.**

### M13: Rust Port + System Tray

Port the Python daemon into Tauri's Rust backend.

- Single binary, no Python dependency
- System tray, launch on login
- Global capture shortcut from anywhere
- Body prompts as native system notifications
- Active window awareness
- Tauri IPC replaces HTTP
- Incremental migration: new features in Rust, port existing endpoints one by one

**Scope: ~8-10h** (can be broken into sub-milestones).
